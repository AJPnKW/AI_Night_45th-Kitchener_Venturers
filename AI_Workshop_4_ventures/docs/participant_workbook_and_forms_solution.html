Participant Workbook and Forms Solution

Recommended end-state solution for individualized participant workflow using an embedded browser workbook engine, Google Forms, Google Sheets, and an optional Apps Script supplement.

What problem this solves

The site now includes an embedded browser workbook engine for the live session, but the project still needs a clear model for when that is enough and when a forms-backed workflow is justified. This page explains that split.

Vision translated into concrete design

What the learner experience should feel likeConcrete implementation meaning
A private workbook that belongs to the learnerEach participant gets a browser workbook flow that saves locally on the same device, plus a prefilled form path if tracked workflow is needed.
A guided notebook, not a blank pageEach activity starts with a selected category, level, starter card, and sentence-frame help.
A learning process, not only a data formEach section asks what changed, what worked, what gap remained, and what to try next.
Support for fast finishersThe workbook flow and optional branching routes quick finishers to Remix or Stretch follow-up work instead of waiting.
Support for stuck learnersGuided sentence stems and narrower follow-up choices reduce overload and blank-page pressure.

Suggested participant workflow

StepExperience
1. Live browser pathParticipant starts from the public site and opens the workbook flow on GitHub Pages.
2. Local continuityThe embedded workbook engine keeps notes in the browser so the learner can leave one step and continue later on the same device.
3. Optional tracked pathIf organizer tracking is needed, the learner instead opens an issued workbook form link.
4. Prompt selectionThe learner chooses from starter prompt options or a guided level path.
5. Note captureThe learner records what they changed, what result they got, and what gap remained.
6. Adaptive progressionBranching or workbook helper logic sends learners to Guided, Build, Remix, or Stretch follow-up items.
7. Completion outputIf the tracked path is used, the response lands in Sheets and can optionally trigger an Apps Script summary email or PDF workflow.

What GitHub Pages should still do

  • Stay as the calm public front door and reference layer.
  • Host the step-by-step workbook engine, exercise cards, examples, KB, and facilitator resources.
  • Provide printable local workbook pages for backup and offline-ish fallback.
  • Link out to the organizer-issued per-participant form when structured tracking is needed.

What the form should capture

  • Participant identifier or alias
  • Device/account readiness
  • Chosen exercise level
  • Selected starter prompt card
  • Prompt revision notes
  • Expectation versus result
  • Reflection answers
  • Need-help or accommodation flags
  • Consent to receive a summary email if that workflow is enabled

Why this is the right answer

This approach is more realistic than trying to force GitHub Pages to act like a full LMS. It uses each tool for what it is good at: Pages for public content and the embedded browser workbook engine, Forms for structured participant flow when tracking is needed, Sheets for visibility, and Apps Script only where actual workflow automation is needed.
