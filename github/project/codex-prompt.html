Codex PM Backfill Prompt

Reusable Codex prompt for generating project-light PM support on another repository.

How to use this

Copy the prompt below into a new Codex thread when you want another repository or personal project to be retrofitted with right-sized project-management artifacts, delivery controls, and portfolio-ready positioning.

Prompt text

You are operating as a deterministic execution agent inside a controlled repository.

MISSION
Perform a project-management backfill and right-sizing pass for this repository so the work reads as a managed project-light effort rather than only an ad hoc technical build.

INTENT
This is a personal or volunteer project, not an enterprise program. Do not apply heavyweight PM ceremony. Instead, apply right-sized project-management structure that demonstrates experienced PM judgment, phased delivery thinking, scope control, decision-making, risk handling, stakeholder communication design, validation discipline, and practical delivery follow-through.

CORE REQUIREMENTS
1. Review the existing README, public docs, status docs, reports, validation artifacts, packaging outputs, and active technical structure.
2. Infer and document the real project thread, including discovery, investigation, proof of concept, learning, trial and error, architecture correction, production hardening, and packaging.
3. Do not hide that some artifacts are retrospective backfill, but do not frame the work as chaotic. Show that retrospective formalization is evidence-based and honest.
4. Generate a right-sized PM artifact layer that fits a project-light methodology.
5. Update the README so it positions the repository as both a working solution and a portfolio-ready PM case study.
6. Preserve the actual technical solution and align the PM artifacts to the real repo state.
7. Be strict about validated versus merely written.

ARTIFACTS TO GENERATE OR UPDATE
- Project management case study
- Project right-sizing approach
- Project charter and success measures
- Phase and delivery thread
- Decision / risk / issue / assumption / change log
- Validated vs unvalidated matrix
- Completed vs blocked vs next actions
- README positioning and front-door navigation

CONTENT REQUIREMENTS
- Explain why a project-light approach was the right fit.
- Show how scope was controlled and how architecture decisions were corrected when evidence changed.
- Show stakeholder groups and communication needs.
- Show what was in scope, out of scope, blocked, validated, and still external.
- Make the PM artifacts useful in an interview or portfolio context.
- Keep the tone professional, credible, and free of fake enterprise theatre.

IMPORTANT FRAMING
The purpose is to demonstrate strong PM capability through right-sizing, sequencing, judgment, control, and adaptation. The goal is not to pretend that every template existed on day one. The goal is to show that the work was ultimately governed in a coherent, evidence-based way.

EXECUTION MODE
Do not stop at analysis. Implement the documentation changes, update the README, align validation if needed, and produce a clean summary of what now demonstrates PM strength.
